# 有給休暇管理アプリ 仕様書

## 1. 概要

社内の有給休暇の付与・取得状況を一元管理するWebアプリケーション。
社員は自分の有給残日数の確認・有給申請を行い、管理者は全社員の有給状況を確認し、申請の承認・却下を行う。

### 対象ユーザー
- **管理者**：人事・労務担当など、全社員の有給情報を管理する立場のユーザー
- **社員**：一般の従業員。自分自身の有給情報のみ扱う

---

## 2. 用語定義

| 用語 | 説明 |
|---|---|
| 全休 | 1日単位の有給休暇 |
| 半休 | 午前 or 午後の半日単位の有給休暇（0.5日として扱う） |
| 時間単位年休 | 労使協定により、年5日分を限度として時間単位で取得できる有給休暇（労働基準法39条4項） |
| 所定労働時間 | 時間単位年休の日数換算に用いる、会社共通の1日あたりの労働時間（本アプリでは8時間固定） |
| 付与 | 労働基準法に基づき、勤続年数に応じて有給日数を社員に付与すること |
| 残日数 | 失効していない有給付与枠（LeaveGrant）ごとの未消化日数を合計した、現在の有効日数（失効済みの付与枠は集計対象から除外する） |
| 繰越 | 前回付与分のうち未消化の日数を、時効消滅前として翌年度に持ち越すこと |
| 時効 | 有給休暇は付与日から2年で権利が消滅する（労働基準法第115条） |

---

## 3. ユーザーロールと権限

| 機能 | 管理者 | 社員 |
|---|---|---|
| 自分の有給情報の閲覧 | ○ | ○ |
| 他社員の有給情報の閲覧 | ○ | × |
| 社員一覧画面へのアクセス | ○ | × |
| 有給の申請 | ○（本人分） | ○（本人分のみ） |
| 有給申請の承認・却下 | ○ | × |
| 社員アカウントの新規登録・編集・退職処理 | ○ | × |
| ログイン | ○ | ○ |

社員は、他の社員のID・URLを直接指定した場合でも自分以外のデータにアクセスできないよう、サーバー側で認可チェックを行う（詳細は「8. 非機能要件」参照）。

管理者の有給申請は本人分のみであり、他の社員の代理申請は行わない。また、管理者は自分自身が申請した有給を自分で承認・却下できない（自己承認禁止）。この制約を成立させるため、管理者アカウントは最低2名の運用を前提とする（詳細は「9. 前提条件・スコープ外事項」参照）。

## 4. 画面仕様

### 4.1 ログイン画面
- メールアドレスとパスワードでログインする
- ログイン成功後、管理者は「社員一覧画面」へ、社員は「自分の詳細画面（マイページ）」へ遷移する

### 4.2 社員一覧画面（管理者専用）
一覧表示項目：
- 氏名
- 有給残日数
- 次回有給付与年月
- 年5日取得義務の状況（5.5参照。達成/未達/要注意のバッジと、複数の未達期間がある場合はその件数）
- 申請中バッジ（有給申請中の社員は行の色分けやバッジで強調表示し、一目で識別できるようにする）

機能：
- 各行をクリックすると、その社員の「社員詳細画面」に遷移する
- 申請中の社員は行の色分けやバッジで強調表示する（承認・却下操作はこの画面では行わず、必ず「社員詳細画面」に遷移して行う）
- 社員アカウントの新規登録画面への導線を設ける

### 4.3 社員詳細画面（社員本人にとってはマイページを兼ねる）
表示項目（一覧画面の項目に加えて）：
- メールアドレス
- 有給取得履歴（日付、単位（全休/午前半休/午後半休/時間単位。時間単位は取得時間も表示）、消化日数の一覧）
- 有給付与履歴（付与日、付与日数、失効予定日）
- 年5日取得義務の状況（5.5参照。基準日・期限・取得済み/取得予定日数）

機能：
- 有給申請フォーム：対象日、単位（全休/午前半休/午後半休/時間単位）を指定して申請する（本人のみ操作可。管理者による代理申請は行わない）。単位が時間単位の場合は取得時間（1〜8時間の整数）もあわせて指定する
  - 同一日・同一区分（全休/午前半休/午後半休/時間単位）の重複申請、および同一日の合計申請日数が1.0日を超える申請はできない（5.4参照）
  - 時間単位年休は義務期間内の合計が40時間（5日相当）を超える申請はできない（5.4参照）
- 申請中の項目はステータス表示（申請中／承認済み／却下／取消済み）で確認できる
- 申請者本人は、「申請中」の申請に限りいつでも自分で取消できる（取消理由は任意入力）
- 申請者本人は、「承認済み」の申請についても、対象日の3日前まで（3日前を含む）であれば自分で取り下げできる（取消理由は任意入力）。取り下げた場合は消化済み日数を残日数に復元する。対象日の3日前を過ぎた承認済み申請は取り下げできない
- 却下理由・取消理由は、申請者本人・管理者の両方が閲覧できる
- 管理者がこの画面を開いている場合、社員情報の編集・退職処理に加え、申請中の案件に対する承認・却下操作を行える
  - 却下時は却下理由を入力できる（任意入力）
  - 承認時に残日数が不足する場合は承認できない（エラーとする）
  - 管理者は、自分自身が申請したものは承認・却下できない（自己承認禁止）
  - 管理者は、対象社員の「年次有給休暇管理簿」をExcel形式で出力できる（管理者限定の機能）
- 社員本人がこの画面を開いている場合は、閲覧・有給申請・「申請中」の取消・「承認済み」の取り下げ（対象日の3日前まで）のみ可能（承認/却下や編集操作は表示されない）

### 4.4 社員管理画面（管理者専用）
- 社員の新規登録（氏名、メールアドレス、初期パスワード、入社日を入力）。登録対象はフルタイム（週の所定労働日数5日、または年間所定労働日数217日以上）勤務者に限る
- 社員情報の編集（氏名、メールアドレスなど）
  - 入社日（hire_date）は、初回の有給付与（LeaveGrant）が1件でも発生した後は編集不可とする（訂正が必要な場合は本仕様のスコープ外）
- 退職処理（アカウントを無効化し、以降ログイン・有給付与を停止する。データ自体は履歴として保持する）
  - 退職処理を行うと、(a) その時点で「申請中」の有給申請は自動的に「却下」扱いとなる、(b) 退職日より後の対象日で「承認済み」の申請は自動的に「取消済み」となる
  - 上記(b)の自動取消では、消化済み日数の残日数への復元は行わない（退職時点の残高を確定値として履歴に保持する）

## 5. 有給付与・失効ロジック

労働基準法に準拠した勤続年数ベースの自動計算を行う（フルタイム正社員を想定。パートタイム等の比例付与は本仕様の対象外）。

労基法上、有給付与には勤続年数に加えて「全労働日の8割以上出勤」という要件があるが、本システムでは出勤率の自動判定は行わない。付与記録（LeaveGrant）に出勤率要件を確認済みであることの記録（確認日時・確認者・情報源）を保持し、実際の出勤率判定は人事側（システム外）で行うことを前提とする。

### 5.1 法定付与日数テーブル（フルタイム・8割出勤要件を満たす場合）

| 勤続期間 | 付与日数 |
|---|---|
| 入社から6ヶ月 | 10日 |
| 1年6ヶ月 | 11日 |
| 2年6ヶ月 | 12日 |
| 3年6ヶ月 | 14日 |
| 4年6ヶ月 | 16日 |
| 5年6ヶ月 | 18日 |
| 6年6ヶ月以上 | 20日（以降毎年20日） |

- 付与日は「入社日から起算した基準日（6ヶ月後、以降1年ごと）」に自動計算される
- 「次回有給付与年月」は入社日を基に自動算出し、一覧・詳細画面に表示する

### 5.2 時効・繰越
- 付与された有給は、付与日から2年で時効消滅する
- 未消化分は翌年度に繰り越されるが、時効（2年）を超えた分は自動的に失効扱いとなる
- 残日数は「失効していない付与枠（LeaveGrant）ごとの未消化日数」を合計して算出する。失効済みの付与枠は集計対象から除外する
- `expire_date`（失効予定日）は「利用可能な最終日」を表す（例：2026-04-01付与分は2028-03-31まで使用可能で、2028-04-01時点で失効済みとして扱う）
- 有給を消化（承認）する際、有効な付与枠が複数ある場合は、失効日（`expire_date`）が近い枠から優先的に消化する（FEFO：First-Expire-First-Out）。同一失効日の場合は付与日（`granted_date`）昇順、次にID昇順で消化順を決定する
- どの付与枠から何日消化したかは、有給申請ごとに「有給消化内訳（LeaveConsumption）」として記録する（7.4参照）

### 5.3 半休の扱い
- 消化日数の計算単位は、全休＝1日、半休（午前/午後）＝0.5日とする
- 残日数は0.5日単位で管理・表示する

### 5.4 時間単位年休の扱い
- 労働基準法39条4項に基づき、時間単位で有給休暇を取得できる（1申請＝1日のうち1〜8時間の整数）
- 所定労働時間は会社共通で8時間固定とする（社員ごとの個別設定は対象外）。消化日数は「取得時間 ÷ 8」で日換算する
- 時間単位年休は同一日に1件までとする（複数の時間帯に分けた申請は対象外）。既存の「同一日の合計申請日数が1.0日を超える申請はできない」ルール（6章）は時間単位にもそのまま適用する（例: 時間単位4時間(0.5日)＋午後半休(0.5日)＝1.0日は許可）
- 時間単位年休の取得は、各義務期間（5.5参照）ごとに合計40時間（5日相当）を上限とする。上限は申請時・承認時の両方でチェックする
- 時間単位で取得した分は、5.5の「年5日取得義務」の取得済み・取得予定のいずれにもカウントしない（労基法上、時間単位年休は当該義務の算定に含まれないため）
- 所定労働時間を8時間以外に変更する場合、または30分等の分単位に対応する場合は、残高計算（FEFO消化）の実装をDecimal演算に見直す必要がある（8時間固定＋整数時間入力の組み合わせでのみ、浮動小数点演算での丸め誤差なしが保証されるため）

### 5.5 年5日取得義務と義務期間
- 年10日以上の有給を付与された社員は、労働基準法の定めにより、基準日から1年以内に5日を取得しなければならない（労基法39条7項）。取得済みの判定は、実際に取得した日（対象日が今日以前の承認済み申請）のみをカウントし、未来日の承認済み申請（取得予定）は含めない
- 各基準日（`annual_auto`の付与日のうち付与日数が10日以上のもの）から起算した1年間（基準日〜基準日+1年-1日）を「義務期間」と呼ぶ。同一社員が複数の義務期間を持つ場合（過年度分が未達のまま新しい義務期間が始まった場合等）は、各義務期間を独立して評価し、最も対応が急がれる期間（期限が近い未達期間を優先）を代表として画面に表示する
- 社員一覧・社員詳細画面に、対象社員の義務期間の状況（達成/未達/要注意）を表示する（4.2/4.3参照）

## 6. 有給申請フロー

1. 社員本人が「社員詳細画面」の申請フォームから対象日・単位（全休/午前半休/午後半休/時間単位）を指定して申請する（管理者による代理申請は行わない）。時間単位を選ぶ場合は取得時間（1〜8時間の整数）もあわせて指定する
2. 申請は1件＝1日（全休/半休/時間単位のいずれか）を基本単位とする。複数日にまたがる休暇は、日ごとに複数件の申請として登録する
   - 同一日・同一区分（全休/午前半休/午後半休/時間単位）の重複申請、および同一日の合計申請日数が1.0日を超える申請はできない（5.4参照。時間単位の場合は取得時間を8時間で日換算した値で判定する）
   - 時間単位年休は、対象日が属する義務期間（5.5参照）内の合計取得時間が40時間（5日相当）を超える申請はできない
   - 複数日をまとめて1回で申請する「期間一括申請」も利用できる（対象は全休のみ、半休・時間単位は対象外）。開始日・終了日を指定し、任意で土日を除外して対象日を確定する（土日除外は入力の便利機能であり、実際の勤務日を保証するものではない。9章参照）。上限は31日。対象範囲内に1件でも既存申請との重複・1.0日超過があれば、その一括申請は1件も作成されない（all-or-nothing）。作成された申請群は共通の識別子（batch_id）を持ち、いずれかが「申請中」の間はまとめて取消・承認・却下できる（一部だけ処理済みの場合は残りのみが対象になる）。まとめて承認・却下・取消は、対象の各申請に既存の単日承認・却下・取消処理を1件ずつ適用するため、途中で残高不足等により失敗した申請があっても、それより前に成功した分はそのまま確定し、残りは処理対象外のまま残る（部分成功を許容する。全体ロールバックするのは作成時のみ）
3. 申請すると「申請中」ステータスとなり、社員一覧画面・詳細画面の両方でバッジ表示される
4. 申請者本人は、「申請中」の間に限り自分の申請を取消できる（ステータスは「取消済み」となり、残日数への影響はない）
5. 管理者が社員詳細画面から承認・却下を行う（一覧画面からの承認・却下操作はできない）
   - 承認：残日数が不足する場合は承認できない（エラー）。時間単位年休の場合は義務期間内の40時間上限も承認時点で再チェックする（エラー）。承認可能な場合、該当分（全休＝1日、半休＝0.5日、時間単位＝取得時間÷8日）を承認時点で即時消化し、消化対象の付与枠をFEFO順（5.2参照）に選択して有給消化内訳（LeaveConsumption）に記録し、有給取得履歴に反映する
   - 却下：ステータスを「却下」にし、残日数には影響しない（却下理由があれば保持し、申請者本人・管理者双方が閲覧できる）
   - 管理者は、自分自身が申請したものは承認・却下できない（自己承認禁止）
6. 承認済み申請のユーザー操作による取消（取り下げ）は、申請者本人が対象日の3日前まで（3日前を含む）に限り可能とする。取り下げるとステータスは「取消済み」となり、消化済み日数（LeaveConsumption）を残日数に復元する。対象日の3日前を過ぎた承認済み申請は取り下げできない。これとは別に、退職処理（4.4）により発生するシステム自動取消は対象日にかかわらず行われ、その場合は消化済み分を残日数に復元せず、退職時点の残高を確定値として扱う
7. 承認・却下・取消（取り下げ）の結果は画面内の表示更新のみで通知する（メール通知は行わない）

## 7. データモデル（概念設計）

### 7.1 社員（User）
| 項目 | 説明 |
|---|---|
| id | 社員ID |
| name | 氏名 |
| email | メールアドレス（ログインID） |
| password_hash | パスワード（ハッシュ化して保存） |
| role | admin / employee |
| hire_date | 入社日 |
| termination_date | 退職日（在職中はnull） |
| status | 在職中 / 退職済み |

残日数・次回付与年月は都度計算して表示する（保存値ではなく、付与履歴・申請履歴から算出する導出値とする）。

hire_date は、初回の有給付与（LeaveGrant）が1件でも発生した後は編集不可とする（訂正が必要な場合は本仕様のスコープ外）。本アプリはフルタイム（週所定労働日数5日、または年間所定労働日数217日以上）勤務者のみを登録対象とする。

### 7.2 有給付与履歴（LeaveGrant）
| 項目 | 説明 |
|---|---|
| id | 付与履歴ID |
| user_id | 対象社員 |
| grant_type | 付与区分（annual_auto：法定自動付与 / special：特別付与） |
| granted_date | 付与日 |
| granted_days | 付与日数 |
| expire_date | 失効予定日（利用可能な最終日。付与日から2年後の前日） |
| attendance_confirmed_at | 出勤率（全労働日の8割以上）要件の確認日時 |
| attendance_confirmed_by | 出勤率要件を確認した担当者 |
| attendance_confirmed_source | 確認方法・情報源（例：勤怠システム名、手動確認など） |

`grant_type` が `annual_auto` の場合、同一 `user_id` + `granted_date` の組み合わせは一意とする（`special` は複数件を許容する）。

### 7.3 有給申請（LeaveRequest）
| 項目 | 説明 |
|---|---|
| id | 申請ID |
| user_id | 申請者 |
| target_date | 対象日 |
| unit | 全休 / 午前半休 / 午後半休 / 時間単位 |
| hours | 時間単位年休の取得時間数（1〜8の整数）。unitが時間単位以外の場合は常にnull |
| status | 申請中 / 承認済み / 却下 / 取消済み |
| requested_at | 申請日時 |
| reviewed_by | 承認・却下を行った管理者 |
| reviewed_at | 承認・却下日時 |
| reject_reason | 却下理由（任意） |
| cancelled_by | 取消を行った主体（申請者本人、または退職処理による自動取消の場合はシステム） |
| cancelled_at | 取消日時 |
| cancel_reason | 取消理由（任意） |
| batch_id | 期間一括申請でまとめて作成された場合の識別子。単日申請の場合は常にnull |

### 7.4 有給消化内訳（LeaveConsumption）
| 項目 | 説明 |
|---|---|
| id | 消化内訳ID |
| leave_request_id | 対象の有給申請 |
| leave_grant_id | 消化元の付与枠（FEFO順に選択、5.2参照） |
| consumed_days | 消化日数（全休＝1、半休＝0.5、時間単位年休＝取得時間÷8の値） |
| cancelled_at | 承認済み申請の自己取消（取り下げ）により残高が復元される際にセットされる日時。退職時の自動取消では意図的にnullのまま残し、この場合残高は復元されない（物理削除せず、監査ログとして保持する） |

## 8. 非機能要件

- パスワードはハッシュ化して保存し、平文では保持しない
- 社員が自分以外のデータにアクセスできないよう、サーバー側（API/バックエンド）で必ず認可チェックを行う（画面制御のみに頼らない）
- 通知は画面内表示（ステータスバッジ等）のみとし、メール等の外部通知は行わない
- ログインセッションが切れた場合は再ログインを求める

## 9. 前提条件・スコープ外事項

以下は今回のスコープを明確にするための仮の設計判断。内容を確認し、認識が異なる場合は修正してください。

- パートタイム従業員の比例付与（週の所定労働日数に応じた按分付与）は対象外とし、フルタイム（週所定労働日数5日、または年間所定労働日数217日以上）正社員のみを対象とする
- 出勤率（全労働日の8割以上出勤）はシステムでは自動判定しない。有給付与履歴（LeaveGrant）に確認記録を保持するのみとし、実際の出勤率判定は人事側（システム外）で行うことを前提とする
- 勤務日適格性（対象日が休日か所定労働日か）は本システムでは保証しない。休日にも有給申請ができてしまう仕様のため、運用側で確認すること（会社カレンダー・勤務日管理は将来的な拡張候補とする）
- ユーザー操作による承認済み申請の取消（取り下げ）は、申請者本人が対象日の3日前まで（3日前を含む）に限り可能とし、消化済み日数は残日数に復元する。退職処理によって発生するシステム自動取消は対象日にかかわらず行われ、復元は行わない
- 入社日（hire_date）の変更は、初回の有給付与後は不可とする（訂正が必要な場合は個別の運用対応とし、本仕様の対象外）
- 管理者は自分自身が申請した有給を承認・却下できない（自己承認禁止）。この制約を成立させるため、管理者アカウントは最低2名の運用を前提とする
- メール通知機能は対象外（将来的な拡張候補）
- 有給の買取制度は対象外
- 1申請＝1日（全休/半休/時間単位のいずれか）を基本単位とする。複数日をまとめて1回で申請する「期間一括申請」(6章参照)は全休のみを対象に実装済み。土日除外は入力の便利機能であり、休日を含め対象日が実際に所定労働日かどうかは保証しない(勤務日適格性は本システムでは保証しないという方針と整合させる)。祝日を考慮した除外は対象外
- 時間単位年休は所定労働時間を会社共通8時間固定・取得時間を整数時間単位とする前提でのみ対応する。所定労働時間を8時間以外にする、または30分等の分単位に対応する場合は残高計算の実装見直し（Decimal演算化）が必要となる（5.4参照）。社員ごとの所定労働時間の個別設定、および同一日に複数の時間帯へ分けた時間単位申請は対象外とする
- 技術スタック（フロントエンド/バックエンドのフレームワーク、データベース等）は本仕様書では規定せず、別途実装フェーズで決定する
