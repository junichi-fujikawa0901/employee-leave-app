# 有給休暇管理アプリ 改善アイデア（TODO / ロードマップ）

現状のコード・仕様書（`spec.md` / `CLAUDE.md`）を読んだ上での改善アイデア一覧。
優先度は「①コンプライアンスの穴 → ②既知の未実装 → ③運用 → ④UX → ⑤技術的足回り」の順でおおむね高い。

---

## ① 法令コンプライアンスの穴（インパクト大）

- [ ] **年5日の取得義務管理（最優先）**
  - 年10日以上付与される労働者は「年5日確実に取得」が会社の義務（2019 働き方改革）。
  - 現仕様にこの概念が無い。付与日（基準日）起点の1年間で5日消化できているかを社員ごとに追跡。
  - 未達社員を管理者ダッシュボードで赤表示・アラート。「あと◯日 / 期限◯月」の残り義務日数を表示。

- [ ] **年次有給休暇管理簿の出力**
  - 労基法で作成・保存が義務（基準日・取得日数・取得時季を記録）。保存期間は原則5年だが、経過措置により当分の間は3年（Codexレビューで指摘・訂正）。
  - 既存データからPDF/Excelで出力。データは揃っているので出力層の追加のみ。

- [ ] **時間単位年休への対応**
  - 労使協定があれば年5日まで時間単位取得が可能。
  - 現状は全休/半休のみ。`LeaveUnit` に時間単位を追加（残高計算・FEFOへの影響大につき設計は慎重に）。

## ② 既知の未実装（CLAUDE.md で宣言済み）

- [ ] **有給自動付与バッチ**
  - 現状は `prisma/seed.ts` の手動生成のみ。
  - 入社日マイルストーン到達で `LeaveGrant` を自動生成する日次ジョブ、または管理者の「付与実行」ボタン＋対象プレビュー。
  - `getNextGrantMilestone` は実装済み。スケジューラ＋冪等性（同一基準日の二重付与防止＝部分ユニーク制約は既存）で完成に近い。

- [ ] **出勤率8割要件の運用フロー**
  - 現状は記録欄があるだけ。
  - 付与予定日前に「出勤率確認待ち」リストを管理者に提示 → 確認者・情報源を入力 → 付与確定、という承認ステップ化。

## ③ 運用を楽にする機能

- [ ] **通知・アラート**（現仕様ではスコープ外）
  - 承認待ちの滞留 / 失効間近（use-it-or-lose-it） / 年5日未達 の3種をメール・Slack通知。
  - まずはアプリ内のお知らせ枠から着手すると軽い。

- [ ] **会社カレンダー / 休日マスタ**
  - 仕様でも「休日にも申請できてしまう」と課題認識済み。
  - 祝日・所定休日マスタを持ち、申請時に「その日は休日」と警告（またはブロック）。

- [ ] **CSV / Excel エクスポート**
  - 給与・勤怠システム連携用に、付与・消化・残高を期間指定でエクスポート。労務担当が最終的に必ず欲しがる。

## ④ UX

- [ ] **管理者ダッシュボード**
  - 承認待ち件数 / 今月失効する日数 / 年5日未達人数 / 全社取得率 を1枚に集約。社員一覧の手前に配置。

- [ ] **チーム休暇カレンダー**
  - 誰がいつ休むかを月表示。休みの集中を管理者が把握できる。

- [ ] **残日数の内訳表示**
  - 合計残日数だけでなく「どの付与枠が・いつ失効するか」を積み上げ表示。社員の早期消化を促す。

- [ ] **期間一括申請**
  - 仕様では将来拡張扱い。連続平日をまとめて選択 → 日ごとの複数申請に自動展開して入力を簡略化。

## ⑤ 技術的な足回り

- [ ] **自動テスト**
  - FEFO消化・残高計算・退職時の自動却下/自動取消は境界条件が多くバグの温床。
  - 純粋関数（`schedule.ts` / `balance.ts` / `request-rules.ts`）に単体テストを厚く。退職処理の統合テストも。

- [ ] **監査ログ**
  - 「誰がいつ承認/却下/退職処理したか」を横断で追える監査ログテーブル。労務トラブル時に強い。
